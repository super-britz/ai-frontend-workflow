# 项目 AI 开发规则

## 协作偏好

- 使用中文沟通。
- 修改前先阅读现有实现、项目文档和相关测试。
- 保持改动聚焦，不扩大无关重构范围。
- 不覆盖用户已有改动；遇到冲突先说明。
- git commit 不添加 `Co-Authored-By` 或任何暴露 AI/Codex 的信息。

## 前端实现规则

- 优先复用现有页面、组件、hooks/composables、services、types 和工具函数。
- 新增 UI 前先搜索已有组件和相似实现。
- 页面逻辑、业务组件、基础组件、接口请求、类型定义保持清晰边界。
- 不随意修改全局样式、主题变量、构建配置、路由结构或公共 API。
- 避免为了单个需求引入新的 UI 库、状态库、日期库或请求库。

## 组件库优先规则

- 基础 UI 默认使用项目现有组件库。
- 不手写已有组件库覆盖的按钮、表单、弹窗、抽屉、表格、分页、上传、通知、菜单、Tab、Tooltip。
- 组件库能力不足时，先封装项目级业务组件，再在页面中使用。
- 新增组件必须说明为什么现有组件无法满足，并补充使用示例或测试。

## 样式与设计 Token

- 样式优先使用项目已有 token、主题变量、工具类或样式约定。
- 不硬编码可复用颜色、字号、间距、圆角、阴影和层级。
- 不通过局部 hack 覆盖组件库内部结构，除非项目已有明确模式。
- UI 文案、按钮、表单项和状态提示必须在移动端和桌面端不溢出、不重叠。

## Figma / 设计还原

- Figma 任务先拆解页面结构、组件映射、状态清单，再实现代码。
- 必须覆盖默认、加载、空、错误、禁用、权限不足、成功等关键状态。
- 设计到代码的映射记录在 `docs/ai/component-map.md`。
- 视觉验收清单记录在 `docs/ai/visual-checklist.md`。
- 设计稿没有说明的交互，优先沿用项目既有模式。

## 测试与验收

- 前端改动后运行项目可用的 lint、typecheck、test、build。
- 新增业务逻辑优先补单元测试。
- 新增组件或页面状态优先补组件测试或关键路径 E2E。
- 有 Playwright/Cypress 时，对核心页面状态做截图或 E2E 验收。
- 无法运行的命令必须在最终回复里说明原因。

## Review 检查

- 是否复用了现有组件和样式系统。
- 是否遗漏 loading、empty、error、permission、disabled 等状态。
- 是否破坏响应式、可访问性、键盘操作或表单校验。
- 是否引入重复组件、重复请求逻辑、重复类型定义。
- 是否修改了无关全局配置或扩大了重构范围。

## 禁止事项

- 不为了快速还原视觉而绕过组件库。
- 不把 mock 数据、调试代码、临时日志留在生产路径。
- 不在没有需求的情况下新增依赖。
- 不删除或回滚用户未要求处理的改动。
- 不提交未验证的 UI 改动。
